## Context

`apps/dashboard` uses Next.js App Router metadata but currently publishes no favicon, touch icon, web app manifest, or brand image assets. There is no existing logo asset to reuse. The dashboard needs one recognizable mark that remains legible at tiny tab sizes and works across desktop browsers, iOS/iPadOS home screens, and Android/Chromium install surfaces.

Constraints:

- Keep implementation within `apps/dashboard`; no API or infrastructure change.
- Use Next.js App Router metadata conventions so links and manifest headers are generated by the framework.
- Keep icon assets static and cacheable.
- Do not claim full PWA behavior; this change supplies identity and install metadata only.

## Goals / Non-Goals

**Goals:**

- Publish a canonical vector mark plus optimized raster variants.
- Cover browser favicon, SVG-capable browsers, Apple touch icons, and Android/Chromium manifest icons.
- Set stable application name, theme color, background color, and maskable-safe icon metadata.
- Validate dimensions, MIME types, alpha behavior, and metadata in automated checks.

**Non-Goals:**

- Add service-worker/offline support or make the dashboard installable as a complete PWA.
- Redesign dashboard branding, wordmark, or in-app logo usage beyond the icon mark.
- Add dark/light alternate icon switching unless later brand requirements demand it.

## Decisions

### Use one source SVG and generated derivatives

Create a square, high-contrast SVG mark as source of truth, then generate derivatives rather than hand-maintaining unrelated images. SVG is used directly where supported; raster files provide predictable rendering where browser or OS icon pipelines do not consume SVG.

Alternative rejected: use only `favicon.ico`. One ICO can support legacy browser tabs, but does not cover Apple touch icons, manifest installs, or modern SVG-capable browsers well.

### Publish a small, explicit icon matrix

Use these outputs:

- `favicon.ico`: 16x16, 32x32, and 48x48 embedded images for broad browser/bookmark compatibility.
- `icon.svg`: scalable browser icon with explicit `viewBox` and no external dependencies.
- `icon-192.png`: 192x192 Android/Chromium manifest icon.
- `icon-512.png`: 512x512 high-density/install icon.
- `icon-512-maskable.png`: 512x512 icon with safe padding for adaptive/maskable surfaces.
- `apple-touch-icon.png`: 180x180 opaque-background iOS/iPadOS home-screen icon.

PNG files use lossless compression and correct `image/png` MIME types. Apple icon receives a deliberate background because iOS applies its own rounded treatment. Maskable output keeps the essential mark inside the central safe zone.

Alternative rejected: ship every historical size such as 57, 72, 76, 114, and 144 pixels. Modern Apple and Chromium metadata no longer require that maintenance burden when 180, 192, and 512 variants exist.

### Use App Router metadata files

Add `app/icon.svg`, `app/favicon.ico`, and `app/apple-icon.png` where Next.js file conventions fit, plus `app/manifest.ts` for typed manifest output. Set explicit `metadata.icons` entries only where needed to expose the SVG and manifest relationship without duplicate conflicting tags.

Alternative rejected: manually inject `<link>` tags in the root layout. Framework metadata avoids ordering and duplication issues and keeps asset discovery conventional.

### Treat manifest as metadata, not PWA runtime

Manifest declares dashboard name, short name, `start_url`, `scope`, `display`, theme/background colors, and both `any` and `maskable` icons. No service worker or offline cache is added.

## Risks / Trade-offs

- [No existing brand source] → Define a minimal Bolt Monitor mark in SVG and keep source file documented so a future brand asset can replace it without changing metadata contracts.
- [Tiny-size detail loss] → Use a simplified mark, test 16x16 rendering, and avoid text or thin strokes in favicon artwork.
- [Maskable cropping] → Keep meaningful geometry within the central safe area and validate the 512px maskable asset visually.
- [Browser cache persistence] → Use stable filenames and rely on normal asset versioning; replacing artwork may require a deployment or query/version strategy if immediate cache invalidation becomes important.
- [False install expectations] → Document and test metadata only; do not add `display: standalone` behavior that implies offline support beyond declared manifest identity.

## Migration Plan

1. Add source SVG and generated icon files.
2. Add Next.js metadata and manifest route.
3. Run dashboard lint, type check, tests, and production build.
4. Verify response content types and icon dimensions locally or in deployment preview.
5. Roll back by removing metadata references and assets; no persisted data migration exists.
